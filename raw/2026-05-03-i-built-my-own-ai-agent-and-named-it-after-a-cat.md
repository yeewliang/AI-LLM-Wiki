*Felix is a self-sufficient, locally-run AI agent I made for myself. Here’s the design behind it*

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*kFVol7f5sz3xlglEuwwowg.png)

Felix, generated by Nano Banana 2

Most of my desk-bound work has nothing to do with writing software (yes, it seems hard to believe but it’s true). Researching, writing, reading slide decks, reviewing meeting notes, organising things, drafting emails, making sense of a long thread of notes. Normal everyday stuff that I end up doing at night because most of my work day is filled with meetings. It’s inevitable that I want an AI agent to help me with this stuff. Something like Claude Code wasn’t the fit. It’s excellent if you want AI assistance writing and debugging code, which isn’t what I needed.

The obvious answer for general agent work was OpenClaw. It runs locally, supports local models, and it’s not particularly hard to install. I tried it and it’s good. But the more I used it, the more I found myself wanting it to work differently. Not wrong, just differently from how I would have built it.

So I built my own. I called it Felix.

## Why Felix?

A quick word on the name, because people ask.

A lot of tooling in the agents world has a ‘claw’ association, because of the most famous one, OpenClaw. But ‘claw’ sounds vaguely threatening, like it might delete your files if you look at it wrong. Felix is a cat. An ordinary, non-threatening, perfectly approachable cat. No one has ever been frightened of Felix or should be.

That’s roughly the design philosophy in miniature, actually.

## What Felix is

Felix is an AI agent that runs entirely on your machine. You install one binary and you’re done. It will self-download a default model (the excellent Gemma4) and you’re off and running.

It can talk to cloud AI providers (Anthropic, OpenAI, Gemini, Qwen, and several others), or it can run entirely with Gemma4 through a bundled copy of Ollama (which is an open-source tool for running AI models on your own hardware). That local option means you can have a working AI agent with no API key, no cloud account, and no internet connection at all, once the initial model has downloaded. The sound of *kaching* ringing at the back of your mind every time the agent spits out something is gone.

All the state Felix accumulates lives in a single directory on your machine, at `~/.felix/`. Conversation history is stored as JSONL, which is one JSON object per line in a plain text file. Memory entries are Markdown files. Configuration is JSON with comments allowed. Skills, the little instruction files that teach the agent how to handle specific tasks, are Markdown with a short header block. You can open any of it in a text editor, copy the whole folder to another machine and pick up exactly where you left off, or delete everything if you want a fresh start. If you know the basic Unix file commands, you already know how to back up, inspect, and operate Felix's state.

The agent has a number of built-in tools it can use to do things in the world. These cover reading and writing files, making targeted in-place edits without rewriting whole documents, running shell commands, fetching web pages and converting them to readable text, searching the web without any API key, driving a headless browser for pages that need JavaScript to load, sending Telegram messages, scheduling recurring jobs that fire on a timer, and loading skills and memories on demand. Images work too. Hand the agent a screenshot or a diagram and it passes the file directly to the model as a visual, not as a description of a visual.

It has a bash tool too. This means the agent can run any command-line program on your machine, whether that’s image processing, video conversion, git operations, file organisation, or anything else with a CLI. Combine that with web fetching and you have something that can pull information from the internet, transform it locally, write the result somewhere, and ping you when it’s done, all without you sitting there supervising it. That’s the loop I actually use Felix for.

Security is built into that loop. The bash tool runs with an allowlist by default, meaning only commands you’ve explicitly approved can execute. The web tools refuse requests to internal network addresses, so the agent can’t be pointed at other devices or services on your local network. File access is contained to your configured workspace and the agent can’t reach outside that boundary regardless of what the model asks for. None of these are settings you have to switch on. They’re the defaults you relax if you need to, not restrictions you bolt on later. There are also no channels, just a concession that allows me to send out Telegram messages.

Felix also supports connecting to external tool servers via MCP (the Model Context Protocol, an emerging standard for wiring AI agents to external services), so you can add capabilities like reading from GitHub, querying your database, checking your calendar, or talking to your own internal services. I have Felix connected to several MCP servers of my own. One queries a personal database, one talks to GitHub, and a couple connect to internal work tools. The agent treats all of these exactly the same way it treats its built-in tools. It doesn’t know, or particularly care, whether a capability comes from an external server or is compiled in. That uniformity is deliberate. All of these, built-in and external, go through the same tool interface, the same permission list, the same audit trail.

Felix also maintains persistent memory across conversations. Every completed session gets ingested into a knowledge graph that surfaces relevant context in future chats, so if you spent an afternoon working through a project, the relevant facts come back automatically next time without you having to re-explain them. A simpler keyword-based memory runs alongside it, always on, requiring no configuration. Skills work in a complementary way. A skill is a Markdown file that tells the agent how to handle a specific domain or task. The default install ships with starter skills for image, video and document tools like ffmpeg, pandoc, and ImageMagick, so the agent arrives knowing how to reason about common command-line operations. You can write your own and drop them into `~/.felix/skills/` (or use the UI).

The memory system is one of those things that seems like a nice-to-have until it quietly saves you. After about a week of using Felix for a project, I noticed it surfacing things from conversations three days earlier that were directly relevant to what I was asking about now, unprompted. I hadn’t thought to reference them myself. It’s not magic (it doesn’t always get it right) but when it works, it’s the closest thing I’ve found to having a collaborator who was in the room for all your previous thinking.

I also added something I called [Cortex](https://github.com/sausheong/cortex), which is a local knowledge graph. Every time you finish a conversation, Felix runs an extraction pass over it and pulls out entities and relationships (people, projects, decisions, technical concepts, named things you discussed) and stores them in a local knowledge graph. On subsequent conversations, Cortex recalls the relevant nodes before the agent starts thinking about your question. It works alongside the keyword search, not instead of it. The two approaches surface different things. Keyword search finds the conversation where you mentioned a specific word. Cortex finds the project that conversation was about, the decision you made in a different conversation two weeks earlier, and the constraint you noted in passing that happens to be relevant now. I wrote Cortex as a separate library before Felix existed, which is partly why it ended up as the long-term memory backbone rather than something bolt-on.

You can also run multiple agents in one Felix install, each with different models, different tool permissions, and different personas (a research agent that can browse the web, a writing assistant with only file access, a task dispatcher that can delegate to the others). They’re all configured the same way, in the same config file, but they can feel completely different, not only because of the models you use but also the personas and capabilities you give them in the system prompt and the tools you allow them to use.

You can also configure sub-agents that other agents can call. I have a setup where a Claude Sonnet agent does the heavy planning and reasoning, and when it needs to execute something repetitive or straightforward, it delegates to a local Gemma4 agent. The local model costs nothing to run and handles execution tasks well. The more expensive model only touches the parts that really need it. You set the whole thing up in the config file, describe what each sub-agent is for, and the supervisor figures out when to call them.

For those who want observability, Felix can export traces, metrics, and structured logs to any OpenTelemetry-compatible collector. It’s off by default, adds no overhead when unconfigured, and you opt in via the config file or standard `OTEL_*` environment variables.

## Felix design principles

I have worked on Felix on and off for the past months, and it has evolved significantly. The software is no longer the same as how it started but having design principles up-front makes things easier when making decisions. I kept the principles straightforward.

### Self-sufficient

One binary, one directory of state, no required network dependency, and the LLM can be local. There is no Felix cloud, no Felix account, no Felix server that could get acquired, or run into a bad database migration, or simply go away one day and take your conversation history with it.

I find this property not much appreciated in most AI tooling. Most software that involves any kind of personalisation, anything that remembers things about you, learns your preferences, accumulates context across sessions, keeps that state somewhere else, on a server you don’t control. With Felix, there’s no version of events where I push a bad server-side update and break your installation. Your sessions are JSONL files on your disk, your memory entries are Markdown files in a folder you can navigate with any file manager, and you can back them up with `rsync`, search them with `grep`, edit them in any text editor. If you want to understand what Felix has remembered about you, you open a folder and read some files.

Plain files compose with the rest of your computer in ways that opaque cloud databases simply don’t. A user who knows `find` and `grep` already knows half of how to operate Felix. The on-disk format is the public API. If you `tar` the `~/.felix/` directory and copy it to another machine of the same operating system, Felix on the new machine picks up exactly where the old one left off. Every conversation, every memory entry, every skill file comes with it. There is no out-of-band state sitting on a server somewhere that the copy doesn't know about, no sync step, no account login required on the other end. You move a folder and you continue your conversation.

That sounds almost mundane when you say it out loud, but think about how many AI tools you’ve used where that would be completely impossible, where your history lives in someone else’s database and your only access to it is through their app, while it still exists. I tarred the directory on my main laptop, unpacked it on a spare machine, ran Felix, and the conversation I’d been having an hour earlier was just there, mid-thought, exactly as I’d left it. No migration, no export format to figure out, no customer support ticket. Plain files don’t have that problem.

### Robust

An agent process that runs for hours or days, accumulates state across crashes and restarts, calls flaky external APIs, and executes shell commands on your machine needs to fail in bounded, predictable, recoverable ways. I developed something close to a rule about this. Every external call gets a deliberate timeout. The knowledge graph recall that surfaces relevant information from past conversations gets 800 milliseconds before the agent loop moves on without it. The bash tool times out at 120 seconds. WebSocket messages are capped at one megabyte. If something breaks, the question should always be “what specifically failed and how do I recover” rather than “why has this been hanging for ten minutes.”

The session format is designed to heal itself after failures. If Felix crashes mid-tool-call, the next time you load that session it detects the orphaned tool calls and injects synthetic error responses so the conversation can continue cleanly. Compaction (the process of summarising long conversation histories to keep the context window manageable) never rewrites history on disk, it only appends a summary entry. If something goes wrong during compaction, you delete that entry and the full raw conversation is still there underneath it.

The same bounded-failure logic applies to external tool servers. If an MCP server starts returning consistent errors, a per-server circuit breaker trips after three consecutive failures and stops calling it. Without something like this, a misbehaving upstream can pull the agent into a self-repair loop. It calls a tool, gets an auth error, calls the tool again, gets the same auth error, burning through LLM tokens until the user manually cancels. The circuit breaker short-circuits that pattern, surfaces a clear message that human intervention is needed, and lets you reset it with the Re-authenticate button already in the chat UI.

It gets more detailed than that. The macOS menu-bar launcher runs the gateway as a separate subprocess in its own process group, and there’s a good reason for it. macOS will occasionally terminate a menu-bar application for reasons entirely outside your control, things like display sleep, memory pressure, or fast user switching. If the gateway lives inside the menu-bar process, a system-level kill takes your active chat session with it. By running the gateway separately, a menu-bar reap just means the tray icon disappears for a moment. Your browser tab keeps streaming, your session keeps running, and relaunching the menu-bar app reattaches to the still-live gateway. Robustness sometimes means thinking carefully about what the operating system itself will do to you.

### Usable out of the box

The default install, with no API keys and no config file edits, has to produce a working agent within a few minutes of installation. When Felix starts and detects no API keys, it quietly configures a local Ollama model (Gemma4, about 9.6 gigabytes) and starts downloading it in the background while you explore the interface. It shows a progress bar. You don’t need to know what Ollama is, or why the model is 9.6 gigabytes, or how any of the plumbing works.

The primary interface is a web page in a browser, not a terminal and not a config file, because that’s what most people (including me) are comfortable with. However if you prefer a command-line interface, `felix chat` is your friend too.

The settings page is a form built over the JSON configuration, so you can change your LLM provider, add MCP servers, or toggle memory settings without touching a file. The config file is there for power users who want it, but it’s not the front door. Most features have to land in the settings UI or work as a sensible default, or they effectively don’t exist for most users. A feature that requires editing a config file does not exist for most people.

To be fair, a 9.6 gigabyte download on first run is real friction. My compromise was to make it transparent, surface the progress prominently, and pre-warm the model once it’s done so the first conversation turn feels instant.

### Secure by default

An agent that can read your files, run shell commands, and make web requests on your behalf can be scarily powerful. That power has to come with sensible defaults, because most people won’t read the security documentation before they start using a tool, and they shouldn’t have to.

Felix binds to localhost only. It’s not reachable from outside your machine unless you explicitly configure it otherwise. No accidental exposure if you forget to set a password. The bash tool starts in allowlist mode rather than full shell access. If you haven’t explicitly approved a command, the agent can’t run it. You can open it up to full access, but that’s a deliberate choice you make, not the default you have to opt out of. The web tools block requests to internal network addresses, so the agent can’t be used to probe other devices on your local network or reach cloud provider metadata services that could expose credentials. File access is contained to your configured workspace. The agent can’t navigate up the directory tree to reach files you didn’t intend it to see. Configuration and session files are written with permissions that only your user account can read.

None of this requires you to do anything. It’s all the starting point. The configuration lets you relax any of it if you genuinely need to, but the defaults protect you from the most common ways an agent with broad capabilities can go wrong.

## Some design worth knowing

There are a few design in Felix’s internals that I particularly liked and want to talk about a bit more.

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*HDfptSWakhw-ZirKi7Gu4Q.png)

Generated by Nano Banana 2

The first is something I was inspired by the Unix design, *everything-is-a-tool*. [I wrote extensively about this before](https://sausheong.com/why-design-is-important-in-engineering-3b8a47d20e7f?sk=66f553d6789950128db6d5faad0d413f). In Felix, tools are the only way the agent can affect the world. File reads, shell commands, web requests, scheduled jobs. MCP servers? They are sources of tools. Skills? They are just content that tell you how to use tools.

All of it goes through the same tool interface, checked against the same permission list, recorded as the same kind of entry in the session history. There’s no parallel concept, no special capability that lets the agent affect the world outside the tool system. Felix does inject some content into the model’s context automatically, things like the index of available skills or relevant facts recalled from the knowledge graph, but those are inputs, not effectors. The agent reads them but it doesn’t act through them.

This sounds obvious until you look at how many agent frameworks have two or three overlapping abstractions for essentially the same idea, “abilities,” “skills,” “actions,” each with slightly different gating semantics. Why go to all that trouble to enforce a single pathway? Having one means one place to check permissions, one place to audit what the agent did, and one concept to explain in the settings UI. It also makes the security model legible. If a tool isn’t on the allowlist, the agent cannot use it. That’s the whole rule. (The only sub-policy worth knowing is on the bash tool, which layers a command-level allowlist on top of the tool allowlist as defense in depth.) Fewer concepts is more usable, simpler to keep robust, and easier to keep secure.

The second is something I call the *index pattern*. The conventional approach in agent systems is to retrieve relevant material, memories, skill instructions, reference documents, and inject them directly into the prompt on every turn, on the assumption that the model might need them. Felix doesn’t do that. Instead, it injects a lightweight index of what’s available (names, tags, a one-line description), and when the agent actually needs a specific memory or skill, it calls a tool to load the full content on demand. The model’s own judgement serves as the retrieval mechanism. This trims roughly five to fifteen kilobytes of speculative content from every single prompt, which matters for response speed and, if you’re paying per token to a cloud provider, noticeably for cost. It also keeps the prompt structure stable across turns, which matters more than it sounds. Hosted AI providers charge meaningfully less, and respond meaningfully faster, when they recognise the beginning of your request from a previous call. Non-deterministic prompt construction quietly costs you money every turn.

To put some numbers on it, hosted AI providers charge roughly ten times less for tokens they’ve already seen at the start of a request, and they respond meaningfully faster because they don’t have to process the whole thing from scratch. A naive agent loop, where skill instructions and memory entries shuffle in and out of the prompt based on whatever seemed relevant that turn, invalidates that cache constantly. The user doesn’t see the mechanics. They just see a bill at the end of the month that’s five or ten times higher than it needed to be, and response times that feel slower than they should. The index pattern is one of the reasons Felix doesn’t have that problem.

The third is something I think of as the *long-running process contract*. Felix is a daemon meant to run for hours or days. If you change your API key in the settings, it should take effect on the next message, not after a restart. The same for editing your tool allowlist, rotating a Telegram bot token, or uploading a new skill. Felix watches the config file and swaps components live when it changes. The LLM clients and permission checker swaps atomically. The skill index refreshes on the next turn. There are two exceptions to this. MCP server connections can’t be gracefully re-established mid-flight, and statically configured cron jobs capture their setup at registration time. The settings UI labels both of these “restart required” to be clear that a restart is necessary.

The fourth is something more mundane but I thought useful. Several places in Felix face a choice between asking you to configure a value twice or *mirroring* it automatically from a related setting. The compaction summariser, which condenses long conversations to keep the context window manageable, uses the same model as your chat agent by default. Cortex, the knowledge graph, uses the same embedder as your memory system. The local Ollama port is injected automatically once the supervisor picks one. You can override any of these explicitly if you need to, but the default is that they mirror each other. The principle is simple. Every value you have to configure twice is a synchronisation bug waiting to happen. You’ll set them differently by accident, update one and forget the other, or wonder later why compaction stopped working when you switched models. Mirroring by default, with an explicit override for the edge cases that genuinely need it, means less configuration that does more.

## Built for myself

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*7oOJhgZ0vDBLFudsCDGamQ.png)

generated by Nano Banana 2

I built Felix primarily for my own use, for tasks that have nothing to do with software development. Research, writing, organising information, automating small recurring parts of my work and personal life. Not a product roadmap or a startup idea, just a tool I needed. Building a tool for myself is quite gratifying. There’s no customer to reassure, no feature list to optimise for adoption metrics, no one to please. It’s all just me. I know it sounds egoistic but I think it’s a sign of times how we can all start developing software that fits us, rather than wait for someone else to build it for us.

The design reflects that because Felix is still evolving as I use it. I’m still constantly developing it. And the direction of that evolution is almost always performance and fit rather than new concepts.

The index pattern came from noticing that response times were slower than they should be and tracing the problem back to speculative content being injected into every prompt. The hot-reload came from frustration at having to restart the process every time I changed an API key. The MCP circuit breaker came from an afternoon where a broken upstream server had Felix spinning in a self-repair loop, burning through tokens while I watched. Each of those was something I ran into personally, doing the same tasks I mentioned at the start. The tool gets faster as I understand where the time goes. Features get added when I feel the absence of them. Things that don’t serve the actual use case get removed or simplified. The codebase is small for what it does, and I want to keep it that way.

If that sounds like the kind of agent you’d find useful, you’re free to download and use it.
