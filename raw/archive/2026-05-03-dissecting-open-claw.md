## A deep dive into OpenClaw system design and architecture

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*dcrBDEIiqskaY7tFsiEAlA.png)

generated by Nano Banana 2

About a year ago, I gave a half-day workshop on AI agents, what it was, and why I thought it was going to change the way we work. The examples I used were mostly agentic workflows, the kind of thing you could model in n8n or in developer-centric frameworks like LangGraph and CrewAI.

The dominant mental model at the time was workflow first. LangGraph drew a clean line between workflows that follow predetermined paths and agents that decide more dynamically, and CrewAI positioned its Flows as the structured control layer with its Crews as teams of agents operating inside that scaffolding. Either way, the agent lived inside something larger.

That mental model was easy to teach because “agentic” really was just an adjective attached to “workflow”. You had a flowchart, or a graph, or a sequence of steps, with an LLM making bounded decisions somewhere inside it. The agent was a participant in the system, not the system itself. You designed the rails, defined the handoffs, and knew more or less how the thing was supposed to proceed. That model still produces a lot of real value, and probably will for years.

Then OpenClaw happened.

OpenClaw does not present as another way to wire agents into workflows, it shifts the centre of gravity. The project describes itself as a local-first gateway and runtime with sessions, channels, tools, voice, skills, browser control, canvas, cron, and companion apps on macOS, iOS and Android. Reading that list, it becomes clear it is not an agent sitting inside a flow, it is an environment in which the agent lives and acts. It sits on your machine, close to your files, your tools, your messages, your browser and your sessions, and the design problem changes accordingly.

In the workflow era the question was orchestration, how to sequence steps, route outputs, retry failures, and stop the LLM from going off the rails. In the agent-environment era the questions are different, and harder. They are about tool permissions, the safe exposure of local resources, session coherence over time, and how to let the agent roam productively without turning the whole machine into a security incident. None of those are workflow problems in the narrow sense. They are runtime problems.

There is one more thing worth mentioning before we start. The maintainer, Peter Steinberger, has openly admitted to generating most of the code with an AI assistant and not reading what comes out. The runtime that hosts the agent was, in a meaningful sense, written by an agent.

That fact also shaped how I read the project. There was little point dissecting code that the maintainer himself does not read line by line, so instead I read the documentation. The markdown files in the repository, `AGENTS.md`, `VISION.md`, `SECURITY.md` and the longer guides under `docs/`, are what define what OpenClaw is supposed to be. The code is the current rendering of those documents, generated and re-generated as the spec moves. This fits the emerging view that in AI-generated codebases the specs become the load-bearing artefact and the code becomes generated output. So this article is a reading of the docs, not of the source.

## Architecture at a glance

Before getting into the design choices, you need a working picture of where the parts live. OpenClaw is not a single binary you point at a chat network. It is a long-running control plane, with a fan of adapters at one edge and a fan of model and tool backends at the other, plus per-device daemons in the background handling anything platform-specific. The README’s line about the gateway being the control plane and the assistant being the product turns out to be the literal shape of the code.

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*O9u_pyqB0vmcJD2Fybe8hg.png)

generated by Nano Banana 2

That control plane is the *gateway*. It lives in `src/gateway/`, dozens of files covering auth, boot, channel-health, chat handling and client bootstrap. Inside it, `src/gateway/protocol/` holds the wire contract that everything else speaks. Channel adapters serialise their messages onto that schema, and a small OpenClaw process running on each user device, called the node-host, deserialises them at the other end. The gateway stays up and routes, and everything else attaches to it.

The node-host is the OpenClaw process that runs locally on your machine. It holds your secrets, bridges to platform-specific things like iMessage, and talks to the central gateway on your behalf. macOS, iOS and Android specific behaviour goes here, alongside the native apps under `apps/` that bind into the system at the platform layer.

Channels are the inbound message sources. The bundled adapters live under `src/channels/` with extension code under `extensions/<name>/`, covering Telegram, WhatsApp, Signal, iMessage, Matrix, Discord, IRC, Nostr, Slack and the rest of the 24. Each adapter normalises whatever its native API hands over into the gateway protocol. Providers are the model backends, also plugins. Skills are markdown documents under `skills/`. Memory is a single backend behind a plugin interface. Sandbox is Docker, used to isolate risky tool execution.

To make all that concrete, picture a WhatsApp text arriving. The WhatsApp adapter normalises the payload onto the protocol and pushes it at the gateway. The gateway routes it to a session. The agent loop runs, reading context, choosing tools, calling the model. A tool call may invoke a markdown skill, reach an MCP server through the bridge under `src/mcp/`, run a command inside the sandbox, or pop something onto the canvas-host. The reply streams back through the same channel adapter and out to the user. One control plane, many ways in.

Anything that talks to the outside world is a plugin, also called an extension in the codebase, and the repository carries 113 of them under `extensions/`. The boundary is enforced by an invariant. `AGENTS.md` spells it out, "core must stay extension-agnostic. Adding a bundled or third-party extension should not require unrelated core edits just to teach core that the extension exists."

Plugins/extensions cross into core only through `openclaw/plugin-sdk/*`, manifest metadata, and documented runtime helpers, never by importing `src/**` directly. The plugin SDK lives at `src/plugin-sdk/` and is the only way in. Skills are a different extensibility route, documents rather than code, in their own `skills/` folder.

State is split deliberately. The gateway holds session state, routing tables, and channel-health and auth bookkeeping. The memory plugin holds long-term knowledge across sessions. Per-device daemons hold local secrets. Channel adapters hold their own auth tokens, each scoped to the channel that needs it. None of this pretends the gateway is a multi-tenant boundary, and the security docs say so plainly. I will come back to that point in the security section.

The shape of the project is also defined by what it refuses to build. `VISION.md` carries an explicit “What We Will Not Merge (For Now)” list, covering nested agent hierarchies, heavy orchestration layers, wrapper channels, and a few other categories the maintainers refuse to absorb. The architecture is shaped as much by those omissions as by the inclusions, and several of the sections ahead come back to that idea.

## Plugin SDK as public contract

OpenClaw has 113 plugins in core, and ClawHub, the project’s external registry, hosts more. That on its own is not unusual for an open-source project of any popularity. What is worth a closer look is what the project does to keep the API each plugin uses from quietly moving every release.

A short detour for context. A *plugin* is third-party code that runs inside the main program. A web browser extension that adds a button to the toolbar, a Photoshop filter, a VS Code extension that teaches the editor a new language are all the same shape. The host program publishes an API the plugin can use. Plugins that stay within the API work. Anything that reaches past the API and into the host’s internals is fragile, because the internals can change at any time and there is no contract that says they will not.

The published API is part of what is called the *plugin SDK*, but an SDK is more than just an API. The API itself is the types and signatures a plugin must match. The SDK adds the runtime helpers a plugin is allowed to call into, the tooling that keeps the API honest, and the guidance that tells plugin authors what is supported. In OpenClaw the SDK lives in `src/plugin-sdk/`, and any plugin that wants to talk to the rest of OpenClaw has to go through it and nothing else.

The interesting part is how the API portion of the SDK is kept honest over time. The SDK ships with a generated snapshot of every type and helper its public API exposes, called the *API baseline*. The build comes with a generator that writes the snapshot to disk and a checker that compares the live API against it. If a contributor changes a published type without regenerating the baseline, CI fails before the change can be merged. The contract is a file on disk with a hash next to it, and the build refuses to pretend nothing has changed.

Many widely-used projects do not work this way. The usual pattern is to ship a product first, watch a community of plugins grow up around the internals, and then try to retrofit a stable API by deprecating things people already depend on.

WordPress’s actions and filters and Node.js’s Stability Index both arrived after the ecosystem had already taken liberties with the internals, and the contract today exists only as a trail of GitHub issues and a release-notes section called “breaking changes”.

A few projects go API-first instead. VS Code published a typed extension API from launch, for instance. OpenClaw belongs in that camp. The SDK was put in place from the start, and the 113 bundled plugins plus everything on ClawHub were grown behind it.

The payoff is plain. Core does not need to learn the name of every channel, provider, speech engine or browser tool, because each sits behind the same published contract. A plugin author can read the SDK and know exactly which parts of the API they are allowed to use. Without the build-time check the API would slowly leak internals every release, the way unmonitored APIs always do.

## Skills are markdown

A `skill` in OpenClaw is a markdown file. Each skill is one document with a small machine-readable header at the top and prose underneath, and that is the entirety of what the agent loads. There is no compiled bundle, no class to instantiate, nothing handed to a function-calling table. The skill is the prose, and the prose is what the model reads.

The header is YAML frontmatter, the same format that powers most static-site generators, so a non-engineer can read it in about ten seconds. The Apple Reminders skill, for example, declares itself like this.

```c
name: apple-reminders
description: Manage Apple Reminders via remindctl CLI.
metadata:
  openclaw:
    os: ["darwin"]
    requires: { bins: ["remindctl"] }
    install:
      - kind: brew
        formula: steipete/tap/remindctl
```

That frontmatter is the entire machine-readable side of the skill. It says which operating system the skill is valid on, which command-line binaries it needs, and how to install them if they are missing. Everything after the frontmatter is plain English. When to use this skill, when not to, what arguments the underlying tool takes. The agent reads the prose and then drives the command-line tool itself using a shell tool it already has, because the skill never claimed to be code in the first place.

The contrast with the conventional approach is what makes this interesting. The usual way to add a capability to an agent is what people in the field call function-calling. You write a class, declare a JSON schema for its inputs and outputs, register the schema into a table the model is told about, and nudge the model to pick the right entry when it needs that capability. The schema constrains the call site precisely, but the human-readable guidance for when to use the tool either gets crammed into the schema’s description field or buried in the system prompt, kept separate from the tool itself.

OpenClaw flips that layering. The skill is itself the context the agent reads when it is deciding whether to pick up the tool. The install plan, the when-to-use guidance, the when-not-to-use guidance and the worked examples all live in the same document. A human can audit the whole thing in a text editor, and a maintainer can change the model’s behaviour by editing the prose, without recompiling anything.

That matters because there are 53 skills in the repo and each one is something a human can read in five minutes and a model can follow without further translation. The Apple Reminders skill drives `remindctl`. The Spotify skill drives whichever of two open-source players is installed. Each skill is a thin layer of prose wrapped around an external command someone else maintains.

Distribution then becomes a publishing problem rather than a release problem. The vision document says the quiet part out loud, that new skills should be published to ClawHub rather than added to core by default. That follows from the format. If a skill is just a markdown file with an install plan, you do not need to ship it inside the gateway binary at all. You publish it to ClawHub, the user installs it on their device, the agent reads it.

ClawHub itself is a real registry, documented at `docs/tools/clawhub.md`, with versioned skill bundles, semver tags, embedding-based search, a `.clawhub/lock.json` lockfile recording what is installed, and a content-hash check that warns the user when local edits diverge from the registry version. The same registry publishes installable plugins, with declared `pluginApi` and `minGatewayVersion` compatibility ranges that the install command verifies before unpacking anything. Together with the plugin SDK from the previous section, that gives OpenClaw two extensibility routes, one for things that need real code and one for everything that does not.

## Channels behind a decomposed contract

The previous section made the case that the plugin SDK is a typed API, sized to the 113 plugins behind it. It did not look at how that API is shaped for one slot in particular, the channel adapter slot, where 24 different chat networks plug in. The channel files in the SDK are where to look.

There are about forty of them, and what jumps out is the naming. Each file is named for one narrow capability. Lifecycle. Status. Streaming. Pairing. Send-result. Mention-gating. Reply-pipeline. Message-actions. Setup. Policy. The contract is not one big interface listing what a channel does. It is twenty or so narrow ones, each describing a single thing a channel might do, and the main contract file is thin, just a list of the names a channel adapter is allowed to import.

The reason that decomposition exists is that chat networks do not agree on what a chat is. iMessage has no real concept of threading. Discord very much does. Telegram’s reactions look nothing like Slack’s. Matrix federates rooms across many servers. IRC channels live behind a leading hash on a single network. Signal groups carry sealed-sender quirks unique to Signal’s privacy model. Nostr is relays and cryptographic signatures with no central anything. None of these networks share a common idea of what a message is once you look past the surface.

A single fat interface called something like `ChatChannel` would force every adapter to fake or no-op every capability listed on it. The iMessage adapter would be permanently lying about threading. The Slack adapter would be permanently no-opping the Nostr-style relay APIs. Any new adapter would arrive with a checklist of stubs to write before it could even compile.

OpenClaw’s answer is to refuse the fat interface. An iMessage adapter implements inbound, send-result, lifecycle and status, and stops there, because that is the honest shape of iMessage. A Discord adapter implements those plus message-actions, mention-gating and threading, because Discord actually has those things. A Telegram adapter picks its own subset. Nothing forces an adapter to advertise a capability it cannot keep, and the gateway can ask, at runtime, whether a given channel supports threading or message actions or pairing, and get an answer that is not a polite fiction.

All twenty-four bundled channels share the contract by sharing only the slots that apply to them. Each capability is a small file. Each adapter implements only the files its network can actually back. That is what makes the same gateway code able to handle a WhatsApp message, a Matrix room update and a Nostr note in the same afternoon without growing a forest of “if this network, ignore that field” branches.

## The agent loop as contract

The *agent loop* is the thing that takes an inbound message, turns it into a model run, lets the model call tools, streams output back, persists what happened, and returns. Every assistant does some version of this. What is interesting in OpenClaw is that the shape of the loop is itself written down as a contract that plugins can intercept.

The lifecycle is documented in `docs/concepts/agent-loop.md`. A turn moves through five rough phases.

1. Intake validates the request and resolves which session it belongs to, returning a run id immediately so the caller is not blocked.
2. Context assembly resolves the model and the auth profile, loads the skills snapshot, builds the system prompt, prepares the workspace, and acquires a write lock on the session.
3. Inference calls the model.
4. Tool execution runs whatever the model picked, sanitises the result, and feeds it back.
5. Streaming pushes assistant deltas, tool events and lifecycle events out to the channel adapter while all this is happening, and persistence writes the new turn into the session transcript before the loop returns.

What makes the loop a contract rather than a method body is the list of named hook points a plugin is allowed to register against. There are roughly a dozen of them. `before_model_resolve` lets a plugin override the provider before the model is resolved. `before_prompt_build` lets a plugin inject extra context, system prompt fragments, or per-turn dynamic text after the session has loaded. `before_tool_call` and `after_tool_call` wrap each tool invocation and can block the call entirely. `before_compaction` and `after_compaction` observe the summarisation passes. `session_start`, `session_end`, `gateway_start` and `gateway_stop` mark lifecycle boundaries, while `message_received`, `message_sending` and `message_sent` cover inbound and outbound mail.

These hooks matter because the loop is the place where the system prompt and the message history get assembled into the byte stream that goes to the model. The cache-stability rules in the next section are guarantees the loop has to keep, and the hooks have to keep them too. A `before_prompt_build` hook returning unsorted entries from a registry would break the cache the same way a registry that did not sort itself would.

There is one more rule worth noting before moving on. Runs are serialised per session key, with one lane per session and an optional global lane on top, so two messages arriving at the same session at the same time queue up rather than racing. That keeps the transcript and tool state consistent without per-session locking inside plugin code.

## Context assembly and cache stability

Of the five phases just described, the one that does the most work behind the scenes is *context assembly*. It is the step that decides which messages to send to the model, in what order, and with what surrounding system prompt. In an agent that has to remain coherent across long conversations on someone’s laptop, that step is not a detail. It is the thing that determines what the model thinks the conversation is.

OpenClaw treats context assembly as a plugin slot too. `docs/concepts/context-engine.md` documents a `ContextEngine` interface with four lifecycle points.

1. Ingest is called when a new message lands in the session and lets the engine store or index it however it wants.
2. Assemble is called before each model run and returns the ordered set of messages that fit within the token budget, optionally with a `systemPromptAddition` prepended to the system prompt.
3. Compact is called when the context window fills up or the user runs `/compact`, and is responsible for summarising older history.
4. After-turn runs once the run finishes, for persistence or background work.

The default engine is called `legacy` and does what OpenClaw has always done. A third-party engine can replace the entire pipeline by registering itself, with the configuration as one line, `plugins.slots.contextEngine: "my-engine"`, and the runtime will resolve to that engine for every turn until the line is removed.

What pulls the section together is that context assembly is constrained by something stricter than a normal plugin contract. It has to keep the prompt cache stable, and `AGENTS.md` gives prompt-cache stability its own top-level section.

A quick concept first. When an assistant talks to a model, it has to send the whole running conversation back every turn, because the model has no memory between turns. The fix is the *prompt cache*. If the start of the new request matches the start of the previous request exactly, character for character, the model can skip re-reading that part and only process the new bit. Anything different in the start, even one character, and the model reads the whole conversation again from scratch. Engineers call that cached start the prefix.

Most projects treat the cache as a performance concern, something to measure and tune when costs creep up. OpenClaw treats it as correctness, and that is the unusual bit. If the prefix changes between turns, the model sees a different conversation, and the assistant behaves differently in ways that look like the model itself getting worse but are really the cache silently breaking because some piece of code went through a list in a different order this morning. There is no error log and no alert. The bytes were valid, just not identical.

To prevent that, `AGENTS.md` writes down five rules, two of which do most of the work. The first is that the order of things in the prefix must be the same every turn. Anywhere a request is assembled from a list of plugins, tools, or files, the code is required to sort the list first and build the request second. JavaScript runtimes do not always traverse a list in the same order, plugin discovery finishes in whatever order disk reads complete, and the operating system returns directory listings in whatever order it feels like. Three fresh chances per turn for the prefix to wobble. Sorting up front is cheap, and debugging "the model got worse this week" is not.

The second rule is about how to shorten the conversation when it gets too long. Cut from the newest end, not the oldest. The prefix lives at the start, so the start has to stay identical turn to turn. Touch the start, lose the cache. Lose the cache, change the model.

The guide also says any change that could affect the cache requires an automated test that proves the prefix stays identical from one turn to the next. Looking through the test suite for one that matches that description does not turn up a clear winner, so I will not name one on a hunch. The point stands either way. The project treats prefix stability as something a test must prove, not something a reviewer eyeballs.

This is where the engine slot and the cache rules close on each other. A plugin engine that does not honour deterministic ordering on assemble will silently break the cache and quietly change the assistant. The contract for replacing the engine is the same contract that demands prefix stability, which means the engine slot is not just an extension point, it is the layer where prefix stability is actually enforced.

Cache misses do not show up as a slower assistant. They show up as a different one.

## The sandbox safety contract

Every other contract in this article is between OpenClaw and a plugin author. The sandbox is different, because it is the contract between OpenClaw and the user’s machine, and that gives it a different shape.

The model is laid out in `docs/gateway/sandboxing.md`. Tools can run inside an isolated backend instead of on the host, with the gateway itself staying on the host. This is opt-in. If sandboxing is off, every tool the agent invokes runs against the user's filesystem, with the user's credentials, in the user's process tree. If it is on, the same tools run inside a container or remote shell that cannot reach the rest of the system unless the user has explicitly bound something through.

The contract has four knobs, each of which is a small but consequential choice.

1. Mode controls when sandboxing is used, off for never, non-main for everything except the user’s main session, all for everything.
2. Scope controls how many containers exist, one per agent, one per session, or one shared between all sandboxed sessions.
3. Backend selects the runtime, with `docker` as the default, `ssh` for offloading to a remote machine, and `openshell` for a managed sandbox service.
4. Workspace access decides what the sandbox can see, none for an isolated scratch directory, ro for the agent's workspace mounted read-only, rw for the same workspace mounted read-write.

Each combination of knobs is a different posture, and the operator picks the one that matches their tolerance for risk.

The Docker backend ships with three images that match three common shapes of work.

1. `Dockerfile.sandbox` is the minimum default at `openclaw-sandbox:bookworm-slim`, deliberately tiny.
2. `Dockerfile.sandbox-common` is the same plus the usual command-line tools a skill might want, `curl`, `jq`, `nodejs`, `python3`, `git`.
3. `Dockerfile.sandbox-browser` is for the headless browser flow with conservative Chromium flags applied by default.

By default the sandbox container has no network at all. Egress has to be turned on explicitly, and even then host networking and namespace-join are blocked unless the user opts into a break-glass override.

The escape hatches are written down with the same care. `tools.elevated` is an explicit way to mark a tool as bypassing the sandbox and running on the host or the node-host instead. Exec approvals, documented at `docs/tools/exec-approvals.md`, layer a per-command consent step on top, with a local JSON file at `~/.openclaw/exec-approvals.json` recording which patterns the user has allowed and at what scope. Approved runs bind canonical execution context, exact argv, env, executable path, and where applicable a single concrete file operand, so an approved command cannot be silently rewritten before execution.

The honesty of the seam is in what it admits. `docs/gateway/sandboxing.md` is plain that this is "not a perfect security boundary" but a way to "materially limit filesystem and process access when the model does something dumb". That is the sandbox's claim, and the docs do not stretch it any further.

## Auth profiles and model failover

Most assistants pick a model, hand the request to a single API key, and surface whatever happens next as the user’s problem. OpenClaw treats provider auth as a separate plane with its own contract and its own resilience model, and the result is a layered, deterministic failover story rather than a “try and pray” one. `docs/concepts/model-failover.md` documents the two-stage pattern. Auth profile rotation within the current provider first, model fallback to the next configured model only after that.

Profiles live on disk at `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`, with runtime routing state in a sibling `auth-state.json`. A profile is either an API key or an OAuth login, both shaped the same way from the runtime's point of view. Multiple profiles per provider are supported by design, so a user with both an OAuth login and an API key for the same provider has two profiles, not one. Profiles are per-agent, which means a multi-agent setup gets per-agent credentials for free, with no extra plumbing.

The rotation order itself is documented and not improvised. If the user has set an explicit `auth.order[provider]`, that wins. Otherwise OpenClaw uses a round-robin order with two keys, OAuth profiles before API keys, and within each type oldest- `lastUsed` first. Profiles in cooldown or marked disabled drop to the end of the list, ordered by soonest expiry. None of that is left to chance.

Within a single session, the chosen profile is then pinned to keep the provider’s prompt cache warm, which ties this section back to the cache-stability discipline from the context-engine section. The same session reuses the same profile until it is reset, until compaction completes, or until the profile lands in cooldown. Switching profiles mid-conversation breaks the provider’s cache the same way reordering the prefix would.

Cooldowns are written down in detail. Rate-limit cooldowns use exponential backoff at 1 minute, 5 minutes, 25 minutes, and a 1-hour cap. Billing failures are treated separately, with a longer 5-hour starting backoff that doubles per failure to a 24-hour cap, on the assumption that “your card was declined” is not transient. The runtime distinguishes provider-busy errors, true rate limits, billing problems, auth failures and timeouts, each with its own classification rules and its own retry shape, and the list of failure phrases the runtime treats as failover-worthy is long enough that it reads almost like a tour of every provider’s worst error message.

Only after every profile in a provider has failed does OpenClaw move to the next model in the configured fallback chain. The fallback selection is persisted to the session before the retry starts, so a live-session reconciliation pass running on another thread cannot snap the model back to the failing primary mid-attempt. If every candidate fails, the runtime throws a `FallbackSummaryError` carrying per-attempt detail and the soonest cooldown expiry, so the user-facing message can say more than "something went wrong".

That is a lot of machinery for one provider call, but the reason it is there is the same reason the cache rules are there. An assistant that runs on someone’s laptop and has to handle a real conversation cannot afford to be brittle about provider behaviour. The contract here is between OpenClaw and the way commercial LLM APIs actually fail, and like every other contract in the codebase, it is named, ordered, and written down.

## Memory and the dreaming pass

OpenClaw treats memory as a plugin slot, with several live backends behind a single contract, plus an opt-in background pass that promotes good short-term notes into durable long-term memory. `docs/concepts/memory.md` is the entry point, and the design holds together because the on-disk shape is plain and human-readable.

Each agent workspace carries a `MEMORY.md` for durable facts that get loaded at the start of every direct-message session, a `memory/YYYY-MM-DD.md` daily file for running observations, and an optional `DREAMS.md` for the consolidation diary. The model only remembers what is on disk, which sounds like a limitation until you realise the trade-off. Markdown rather than a binary store means the user can open these files, edit them, version them in git, and share them between devices without a migration step.

Behind that, the memory plugin slot accepts more than one implementation. The default `memory-core` is a SQLite backend with keyword search, vector similarity and hybrid search out of the box. `memory-qmd` is a local-first sidecar with reranking, query expansion, and the ability to index directories outside the workspace. `memory-honcho` is a cloud-backed store with cross-session user modelling. `memory-wiki` is a separate companion plugin that compiles durable knowledge into a wiki vault with structured claims, evidence, contradiction tracking and generated dashboards. Each plugs into the same contract, the same `memory_search` and `memory_get` tools, the same session bootstrap, so the user picks one and the agent does not need to know which.

Search uses hybrid retrieval when an embedding provider is available, with vector similarity for meaning plus keyword matching for exact terms like identifiers and code symbols. OpenClaw auto-detects the provider from whichever API keys are already configured, so there is no separate embedding service to wire up.

The piece that earns a longer second look is dreaming, documented at `docs/concepts/dreaming.md`. Opt-in by default. When enabled, a recurring background sweep runs three phases in sequence. Light phase ingests recent daily notes and recall traces, dedupes them, stages candidates, and never writes to durable memory. Deep phase ranks the staged candidates with weighted scoring and threshold gates, then promotes the survivors into `MEMORY.md`. REM phase extracts themes and reflective signals that feed the next deep ranking. The Dream Diary at `DREAMS.md` is the human-readable summary of all of this, reviewable at any time.

Two safety rails are worth flagging. Long-term promotion writes to `MEMORY.md` only, and no other path mutates that file without going through the deep phase. And before compaction summarises a long conversation, the loop runs a silent turn that nudges the agent to save important facts to memory first, so context loss during summarisation is opt-out rather than opt-in.

## MCP is bridged, not first-class

I said earlier that the MCP bridge lives under its own folder. You might think this is a real Model Context Protocol runtime, the kind a serious assistant project would build to keep MCP traffic running in-process and under its own logging and policy controls. The directory does not contain one, and that absence turns out to be the design.

A short detour for the unfamiliar reader. MCP, the Model Context Protocol, is an open standard that lets agents call out to external tool servers, the way a web browser uses HTTP to talk to web servers. An MCP server might wrap a calendar, a database, a ticketing system, or a code repository. The protocol is young, popular, and revising itself every few months as the shape of agent-tool wiring keeps changing. Most ambitious assistant projects have raced to ship their own first-party MCP runtime, because being able to say “we are MCP-native” is good positioning.

OpenClaw did not. The MCP folder turns out to hold nine files, all bridging shims and tool-serving wires, the plumbing that lets MCP-shaped traffic flow into and out of the gateway. There is no first-party MCP server runtime or transport stack carrying the spec on its own shoulders. The actual MCP integration is delegated to a separate project called *mcporter*, and the vision document is up-front about it.

That decision is written down twice. Once in the section on MCP support, which says OpenClaw supports MCP through mcporter so core can stay lean and the user can add or change MCP servers without restarting the gateway. Again in the explicit “what we will not merge” list, which calls out a first-class MCP runtime in core, while mcporter already provides the integration path, as something the project deliberately refuses to absorb.

The same discipline-of-scope thinking that keeps core extension-agnostic is doing the work here, applied to a protocol that is moving fast. A first-class MCP runtime inside core would mean every MCP spec change becomes a core release, every transport-layer fix becomes a gateway redeploy, and any security finding in the MCP stack lands inside the gateway’s blast radius. Bridging means MCP changes can land in mcporter without anyone touching the gateway code, and the gateway’s stability and security exposure stays small.

The boundary is honest in the other direction too. If a feature is missing, the vision document tells users to open an issue at mcporter, not at OpenClaw, which means the maintainers are not pretending to own a stack they have chosen not to ship.

The bridge model is not free. A user wiring up an MCP server has to install mcporter as a separate moving part and trust the same person shipping OpenClaw to ship the bridge in a coordinated fashion. What you get back is a core that does not break each time MCP does, a security review that does not have to relitigate the MCP transport layer, and a clean line on the architecture diagram with a named project on the other side of it.

## OpenClaw also speaks ACP

Some background first. ACP, the Agent Client Protocol, is a young open standard that lets a text editor drive an AI agent from inside the IDE, the way it might drive a debugger or a language server. The editor opens an agent panel, the user types a prompt, and the editor speaks ACP to whatever agent is plugged in to handle that prompt. Most agent projects ship in chat clients first and worry about IDE integration later, if at all.

OpenClaw speaks ACP. After everything else this article has covered, the natural reading would be that the gateway routes to WhatsApp, Telegram, Signal and the rest, and any IDE story is left to the model providers. The ACP folder says otherwise. Inside it sits a small bridge, a translator, a secret-file helper and a server, all wired up against the published SDK for ACP. Two ways in, one gateway underneath.

The shape of the bridge is straightforward. The editor launches OpenClaw in ACP mode and connects to it the way a parent process connects to a child, over the simple input and output streams every command-line tool uses. The bridge accepts the editor’s session, hands it to a translator, and the translator maps that editor session to a gateway session key. From there the same gateway that handles a WhatsApp message handles the editor’s prompt. Same routing. Same provider plugins. Same skills. Same MCP bridge from the previous section.

The bridge does not implement the whole protocol, and the docs are honest about it. Initialise, newSession, prompt and cancel work. LoadSession is partial. Per-session MCP servers are not supported. This is a gateway-backed ACP bridge, not a full ACP-native editor runtime, designed to route IDE prompts into an existing gateway session with predictable session mapping and basic streaming updates.

What makes the section worth writing is the architectural payoff. The same gateway that already speaks twenty-four chat channels can now also speak to an editor, because both flows feed the gateway messages of the same eventual shape. The bridge stays small because it does not have to reinvent routing, history or tool dispatch.

## Multi-agent, but flat

There is a tension in the docs that is worth flagging before it confuses anyone. `VISION.md` 's "What We Will Not Merge" list calls out "Agent-hierarchy frameworks (manager-of-managers / nested planner trees) as a default architecture" alongside the MCP runtime refusal. At the same time, `docs/concepts/multi-agent.md` describes hosting multiple isolated agents in one gateway, and `docs/tools/subagents.md` describes spawning sub-agents from any running session. That looks like a contradiction at first glance. It is not, and the resolution is in the shape.

The first piece is multi-agent routing. One gateway can host many agents side by side, each with its own workspace, its own state directory, its own session store and its own auth profiles. Inbound messages are routed to a specific agent through a list of bindings. A binding matches on channel, account, peer, and a handful of platform-specific keys, with the most specific match winning. One Discord bot can route to a coding agent, another to a personal agent, both running in the same gateway process. WhatsApp DMs from one number can route to one agent and DMs from another to a second. Each agent is a separately scoped brain.

The second piece is per-agent persona. Each agent’s workspace carries a `SOUL.md`, documented separately at `docs/concepts/soul.md`, which OpenClaw injects into the system prompt for sessions on that agent. The Soul guide is direct about what belongs there, tone, opinions, brevity, humour, default level of bluntness, and what does not, life stories, security policy dumps, generic corporate sludge. Two agents on one gateway can therefore sound nothing like each other because each one's Soul file is its own.

The third piece resolves the apparent contradiction. `sessions_spawn` lets a running agent kick off background work in a separate session, with a default `maxSpawnDepth` of one. Recursion is opt-in. The orchestrator pattern, where one sub-agent spawns its own sub-sub-agents, is allowed only if the operator sets `maxSpawnDepth` to two, and never beyond depth two as the default. The runtime supports nested agents, but it does not assume nested agents are how a user should think about agents.

So the picture is multi-agent without hierarchy. Multiple personas in one gateway, routed by binding rather than by management. Sub-agents available for parallel work, capped at one level deep by default. The “manager-of-managers” pattern that VISION refuses is the case where one agent is given a tool whose job is to plan and dispatch other agents in a recursive tree. That is what OpenClaw refuses as the default, and the rest of the multi-agent surface fits underneath that refusal.

## Cron jobs in isolated agents

Most schedulers in assistant projects are callbacks. The cron tick fires, a function runs, the result lands in the same session the user is already chatting in. OpenClaw does not work that way, which is a surprise.

`src/cron/isolated-agent/` is a directory of about forty files describing a scheduling layer where each cron tick spawns its own agent, with its own session identity, its own auth profile, its own delivery lane and its own tool policy. The cron job is not a callback into the user's session. It is a fresh actor that runs once and exits.

The shape of the run is in `src/cron/isolated-agent/run.ts`. When a cron job fires, OpenClaw resolves the cron-specific session key, loads the skills snapshot for the agent the job is bound to, picks a model and a thinking level for that job, builds a cron-specific defaults config, prepares the workspace, and runs an agent turn end-to-end against that ephemeral session. If the resulting reply is just a heartbeat acknowledgement, the delivery layer notices and suppresses the outbound message rather than spamming the user. Otherwise the result is delivered to whichever channel the cron job was configured to publish to.

The supporting files in the same folder are specific and worth naming. `run-auth-profile.runtime.ts` resolves which auth profile this job's agent should use, separately from the main session. `run-model-selection.runtime.ts` lets a job pin a different model than the agent's default. `run-fallback-policy.ts` decides what happens when the chosen model fails. `run.sandbox-config-preserved.test.ts` exists specifically to prove that a job's sandbox configuration does not leak from one tick to the next. `subagent-followup.runtime.ts` handles the case where a cron-spawned agent itself spawned a sub-agent that finished after the parent returned.

The reason this is interesting is the boundary it draws. A scheduled job is not allowed to mutate the user’s main session by accident. It cannot pick up the main session’s auth profile by mistake. It cannot inherit the main session’s tool policy. Its sandbox configuration is reasserted every run. If something goes wrong, the blast radius is the cron job’s session, not the user’s chat history, and the runtime makes a fresh agent for each tick precisely so that boundary stays clean.

## What this article didn’t cover

A few things did not make it into this dissection, and they are worth flagging so the article does not pretend otherwise.

The voice and realtime audio pipeline under `src/realtime-voice/` and `src/realtime-transcription/` is a full bidirectional voice stack, more substantial than a typical speech engine wrapper.

The agent-driven browser tool with its own sandbox lifecycle and CDP plumbing under `src/proxy-capture/` is its own design problem.

The auto-reply layer at `src/auto-reply/` sits between the channel adapters and the agent loop, doing trigger detection, command parsing and group-activation gating.

Canvas and `src/canvas-host/a2ui/` is the agent-drawn UI surface.

Pairing, Bonjour discovery and multi-gateway routing are a distinctive local-network story.

The native apps under `apps/android`, `apps/ios` and `apps/macos` were named earlier in this article but I didn’t talk too much about it.

Each is a section's worth of material on its own. The pattern is the same in every case. Named contract, narrow boundary, documented behaviour. Pick one and read its `docs/` entry, then read the matching folder under `src/`, and you will see the same shape.

## Security and trust model

Everything above adds up to a runtime that does a lot on the user’s machine, which raises the obvious question of what happens when something reaches the gateway that should not. Before the closing, here is a frank look at what a reader is signing up for when they run this thing.

Start with the trust model, because everything else is read against it. `SECURITY.md` puts it like this.

```c
## Operator Trust Model (Important)

OpenClaw does **not** model one gateway as a multi-tenant, adversarial 
user boundary.
- Authenticated Gateway callers are treated as trusted operators for 
  that gateway instance.
- Direct localhost/loopback Control UI and Gateway WebSocket sessions 
  authenticated with the shared gateway secret (\`token\` / \`password\`) 
  are in that same trusted-operator bucket. Local auto-paired device 
  sessions on that path are expected to retain full localhost operator 
  capability; they do not create a separate \`operator.write\` vs 
  \`operator.admin\` security boundary.
- The HTTP compatibility endpoints (\`POST /v1/chat/completions\`, 
  \`POST /v1/responses\`) and direct tool endpoint (\`POST /tools/invoke\`) 
  are in that same trusted-operator bucket. Passing Gateway bearer auth 
  there is equivalent to operator access for that gateway; they do not 
  implement a narrower \`operator.write\` vs \`operator.admin\` trust split.
```

In plain English, the gateway treats anyone holding the shared password as the operator, full stop.

“Loopback” and “localhost” are network-speak for “the same machine the gateway runs on”, a “WebSocket session” is a long-lived two-way connection an app keeps open to the gateway, and “bearer auth” means presenting that shared secret as your credential. The `operator.write` and `operator.admin` labels are finer-grained roles a multi-user system might offer, and OpenClaw deliberately does not split them.

That is a deliberate, narrow trust model, and it is the lens for the rest of this section. The point is not that OpenClaw is insecure. The point is that OpenClaw is honest about which boundaries it defends and which it does not, and a user who runs the gateway is responsible for knowing the difference.

So, the entry points, in roughly the order a reader would meet them.

Prompt injection from chat is the first. Anyone who can message you on a wired-up channel can try to inject. `SECURITY.md` lists "Prompt-injection-only chains without a boundary bypass (prompt injection is out of scope)" as a common false-positive pattern. That is not being dismissive but being honest about the limits of LLMs. There is no current training trick that reliably defends a model against a maliciously crafted message inside a conversation it is supposed to read.

The plugin supply chain is the second. The 113 plugins in core sit behind the plugin SDK, third-party plugins are encouraged, and skills are published to ClawHub by design. Each plugin you install is code that runs inside your gateway with your tokens. The same `SECURITY.md` lists "Reports that only show a malicious plugin executing privileged actions after a trusted operator installs/enables it" as a false positive. Once you install something, it is the operator.

Credential sprawl across channels is the third. The channel contract turned twenty-four chat networks into one runtime, and a side effect is that the bot tokens for every channel you wire up live in the same place. One compromise, many tokens.

Skills drive arbitrary CLIs, which is the point of the markdown-skills design. The Apple Reminders skill exists to drive `remindctl`, the Spotify one drives `spogo`, and a skill's `requires.bins` block is exactly a list of binaries the agent will reach for. There is real, partial mitigation in the form of `Dockerfile.sandbox`, `Dockerfile.sandbox-browser` and `Dockerfile.sandbox-common`, which let you run the riskier flows inside containers. Partial because they are opt-in and not every skill flow runs through them.

The gateway is also a long-running daemon. Always-on, attack surface constant.

Finally, HTTP and ACP endpoints are full-operator. `SECURITY.md` is explicit on the false-positive list that the Gateway HTTP compatibility endpoints (`POST /v1/chat/completions`, `POST /v1/responses`) authenticate the shared bearer secret and are full operator-access surfaces, not per-user/per-scope boundaries. The ACP bridge sits in the same bucket once authenticated. Expose either, expose the operator.

To be fair, OpenClaw does a lot of the work it can do. `src/security/` holds roughly 80 entries, around 50 of them named `audit-*`, a real audit framework rather than a few sanity checks. `INCIDENT_RESPONSE.md` defines a formal severity guide whose Critical row reads "Package/release/repository compromise, active exploitation, or unauthenticated trust-boundary bypass with high-impact control or data exposure." There is a separate `openclaw/trust` repo for the threat model, and a named Security & Trust lead. The project is taking this as seriously as a small team can.

Which leaves what a reader can actually do. Treat the gateway as a single-user system and run it on devices only you use. Be selective about plugins, stick to core where you can, and read the source on third-party ones before installing because a plugin is code with your tokens. Limit channels to the ones you actually use, because each one is a credential and an inbound entry point. Keep up with releases and watch the project’s GitHub Security Advisories, because the project does publish them when something matters.

This is the price of an assistant that actually does things. The architecture is honest about its entry points, and that honesty is what makes the trust model usable.

## Closing

Step back, and a single shape comes into view across everything this article has covered. The thing OpenClaw does well is name the seams. Each part of the runtime that could have been a vague convention is instead a contract on disk. The plugin SDK has a generated baseline that breaks the build when it drifts. Prompt-cache stability is promoted to a tested invariant in `AGENTS.md`. The sandbox has four named axes and three reference Dockerfiles. Auth profiles rotate by a written order with classified cooldowns. MCP is delegated to `mcporter`, with the refusal noted twice in `VISION.md`. Every section in this article was a different version of the same move.

Treat each as a published contract and the rest of the architecture follows from it. Core stays small because every other thing is somebody’s plugin or somebody’s protocol. The product is not the assistant. The product is the integration boundary, and it’s all there in the docs.

That discipline is what the workflow framing did not prepare us for. A flowchart-shaped LLM application can get away without it because the graph is the constraint, the steps are the rails, and the model only ever decides what to do inside a box you drew. An agent-environment has no box. The runtime is the operator’s machine, the entry points are live chat networks and a working editor, the tools touch files on disk, and the agent has initiative across all of it. There is no graph holding it together. There are only the seams.

A year from now the code will look different again, because the maintainer will have regenerated half of it. The seams will still be where the docs say they are. That is what makes a project written this way legible, and it is what makes a personal AI assistant something a small team can actually keep shipping.
